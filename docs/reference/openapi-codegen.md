# OpenAPI codegen

Journey regenerates two files from your OpenAPI spec: `generated/endpoints.ts` (one `EndpointRef<T>` per operation) and `generated/models.ts` (`openapi-typescript` schema types). Regeneration is **non-destructive** — nothing outside `generated/` is touched.

## What gets generated

### `generated/models.ts`

Emitted by `openapi-typescript`. Exports:

- `paths` — mirrors the `paths:` tree from the spec, one key per URL template, one sub-key per HTTP method. Mostly internal.
- One interface per named schema under `components.schemas`.
- `operations` — per-operation parameter and response types.

File header:

```ts
// AUTO-GENERATED BY @journey/codegen — do not edit by hand.
```

### `generated/endpoints.ts`

Emitted by Journey's own codegen. Exports a single `endpoints` object — one property per operation, keyed by the operation's derived name (from `operationId` if present, else a stable name derived from method + path). Values are `EndpointRef<ResponseType>` with `method`, `path`, and `operationId`.

```ts
export const endpoints = {
  login: {
    method: "POST",
    path: "/auth/login",
    operationId: "login",
  } as unknown as EndpointRef<LoginResponse>,

  getPetById: {
    method: "GET",
    path: "/pet/{id}",
    operationId: "getPetById",
  } as unknown as EndpointRef<Pet>,

  // …
} as const;
```

The response type is a phantom brand — at runtime `endpoints.getPetById` is a plain object with just the three properties.

## When to regenerate

Run `journey generate` whenever the spec changes. The GUI surfaces drift (endpoints added / removed compared to the generated file) in three places:

- Sidebar badge on **Spec diff**.
- Card on the **Overview** page.
- Full diff on the **Spec diff** page, with a **Run journey generate** button.

There is no auto-regeneration — regeneration is always explicit so it shows up in `git diff`.

## What happens if an operation disappears

The generated `endpoints` object loses the key. Any `.journey.ts` referencing it stops type-checking:

```
Property 'getOldPet' does not exist on type '{ login: EndpointRef<…>; … }'
```

Fix up the journey (rename or replace), or — if you need to call a removed operation — switch it to a [descriptor endpoint](../guide/writing-journeys/endpoints) until the spec catches up.

## Invocation

```sh
journey init <dir> --spec ./openapi.yaml    # generates once during scaffold
journey generate                             # regenerates in place
```

`journey generate` reads `journey.config.json`, resolves `spec` to an absolute path, and rewrites `generated/`. It also performs a read on `journeysDir` as a sanity check — but writes nothing there.

## Programmatic access

```ts
import { generate } from "@journey/codegen";

const result = await generate({
  specPath: "./openapi.yaml",
  outDir: "./generated",
});
// result.operationCount, result.modelsPath, result.endpointsPath
```

## Non-destructive guarantee

The codegen runner only writes to:

- `<outDir>/models.ts`
- `<outDir>/endpoints.ts`

It does not read or modify `journeys/`, `environments/`, or `.journey/cache/`. If you want to preserve hand-written shims alongside generated code, put them in a sibling directory (e.g. `endpoints-ext/`) and import from both.

## Warnings

The codegen runner emits non-fatal warnings to stderr; the run still completes.

### Mixed path prefixes

If most operations sit under one top-level segment but a small minority sit under another (≥ 80% / ≤ 20% split with at least 5 operations total), the runner emits:

```
journey: warning — 1 operation(s) use prefix '/api' while 30 use '/v1'
```

This catches the typical typo case — one operation written as `/api/v1/foo` while every other operation is under `/v1/...` — without flagging legitimately multi-prefix specs that use a balanced split. Combined with a `baseUrl` ending in `/api`, the odd one out would resolve to `/api/api/v1/foo`; the warning steers you to the spec before you notice the 404 in a journey run.

Available programmatically as `findPrefixOutliers(operations)` from `@journey/codegen` — returns `null` when there's nothing to flag, otherwise a `PrefixLintFinding` with `majority`, `minority`, and `message`.

## Drift detection

The `journey serve` backend exposes a spec-drift endpoint used by the GUI. It parses the current `openapi.yaml`, compares operation IDs against the symbols exported from `generated/endpoints.ts`, and returns two lists: added (spec has, generated doesn't) and removed (generated has, spec doesn't). The same detection surfaces in the sidebar badge, Overview card, and Spec diff page.
